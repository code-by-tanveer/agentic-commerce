State residency:

| What                         | Where                                         | Lifetime           |
|------------------------------|-----------------------------------------------|--------------------|
| LLM model weights            | Groq Cloud                                    | n/a                |
| Conversation messages        | SQLite `messages` table                       | per session row    |
| Preferences (About You card) | SQLite `preferences` table                    | per session, edit  |
| Shortlists (Love/Maybe/Skip) | SQLite `shortlists` + `sessionStorage` mirror | per session        |
| Saved outfits                | SQLite `saved_outfits`                        | per session        |
| Catalog responses            | In-proc LRU cache (15 min TTL, 500 entries)   | until restart      |
| Groq access key              | Backend env var only — **never** ships to FE  | process            |
| User-uploaded images         | Disk (`UPLOAD_DIR`) or S3 (env-toggle)        | 24h auto-purge     |
| OG share images              | Next.js `@vercel/og` runtime                  | regenerated on hit |

The agent loop lives **inside** the Fastify process — not split into a worker. The loop is short (≤4 turns), bounded by Groq streaming latency, and SSE is naturally long-lived. Splitting it into a queue adds latency and operational surface area for no gain at our current scale.

## 2. Data flow for a single user message

```
1. User types in InputBar → useConversation.send(text)
2. Frontend POSTs to /api/chat
     headers: { accept: 'text/event-stream' }
     body:    { sessionId, messages: [...history, {role:'user', content:text}] }
   using @microsoft/fetch-event-source (because POST + SSE; EventSource is GET-only).

3. Fastify chat.ts route:
     a. validates body with Zod
     b. opens SSE: sets Content-Type: text/event-stream, no-cache, x-accel-buffering: no
     c. derives sessionId (cookie or body), upserts session row
     d. loads preferences from SQLite for this session
     e. constructs system prompt = base + preference snapshot + intent rubric
     f. hands (history, system, tools[]) to services/agent.ts

4. services/agent.ts run loop (per turn, max 4):
     a. groqClient.streamChatCompletion({ model, messages, tools, tool_choice:'auto' })
     b. iterate the OpenAI-shape stream:
          - chunk.choices[0].delta.content → emit { type:'text_delta', text }
          - chunk.choices[0].delta.tool_calls → accumulate by index until "finish_reason:tool_calls"
     c. on tool_calls completion:
          - emit { type:'tool_status', toolCallId, name, args, status:'running' }
          - Promise.all(tool_calls.map(tc => toolRegistry.dispatch(tc.name, parse(tc.args), ctx)))
          - on each settle: emit a typed result event (products | comparison | outfit | moodboard
            | preference_update | reasoning_chip), and { tool_status, status:'done' | 'error' }
          - append tool messages to the running history
          - loop (next turn)
     d. on finish_reason='stop': emit { type:'done' }, close stream
     e. on error/timeout/max-turns: emit { type:'error', code, message }, close

5. Frontend lib/stream.ts yields each parsed event. useConversation reducer:
     - 'text_delta'        → appends to current assistant message's text block
     - 'products'          → pushes a 'products' sub-block onto the assistant message
     - 'comparison'        → pushes a 'comparison' sub-block
     - 'outfit'            → pushes an 'outfit' sub-block
     - 'moodboard'         → pushes a 'moodboard' sub-block (image-extracted attrs)
     - 'tool_status'       → updates the inline status indicator state
     - 'reasoning_chip'    → attached server-side to products (delivered with the products event)
     - 'preference_update' → mutates the PreferencesCard state, triggers a toast
     - 'error'             → flips message to error state, surfaces graceful retry
     - 'done'              → finalizes the message, persists last delta to SQLite via /api/session

6. MessageRenderer walks sub-blocks in order, rendering each into its component.
   The conversation reads as a stream of *components*, not a wall of text.
```

The agent loop is **deterministic in its event protocol** — every tool maps to exactly one frontend block type. The LLM is free in *what* and *when*, not in *how* the UI receives it.

## 4. Data model

SQLite via `better-sqlite3`. WAL mode. Single file at `data/agentic.db` (path from `DB_PATH`). Foreign keys on. Timestamps stored as ISO-8601 strings for human readability — we are not at the scale where epoch ints matter.

```sql
-- 0001_init.sql

CREATE TABLE sessions (
  id            TEXT PRIMARY KEY,          -- nanoid, 21 chars
  created_at    TEXT NOT NULL,
  updated_at    TEXT NOT NULL,
  user_agent    TEXT,                      -- for abuse forensics
  ip_hash       TEXT,                      -- sha256(ip + salt), never raw IP
  view_mode     TEXT NOT NULL DEFAULT 'list', -- 'list' | 'collage'   (UX move #1)
  summary_blob  TEXT                       -- serialized final lookbook (Phase D / move #7)
);

CREATE INDEX idx_sessions_updated ON sessions(updated_at);

CREATE TABLE preferences (
  session_id    TEXT NOT NULL REFERENCES sessions(id) ON DELETE CASCADE,
  key           TEXT NOT NULL,             -- 'size', 'budget', 'ships_from',
                                           -- 'palette', 'ethics', 'shipping_speed'
  value         TEXT NOT NULL,             -- JSON-encoded; allows {min,max} for budget etc.
  source        TEXT NOT NULL,             -- 'user' | 'inferred' | 'agent'  (UX move #3 transparency)
  updated_at    TEXT NOT NULL,
  PRIMARY KEY (session_id, key)
);

CREATE TABLE messages (
  id            TEXT PRIMARY KEY,
  session_id    TEXT NOT NULL REFERENCES sessions(id) ON DELETE CASCADE,
  role          TEXT NOT NULL,             -- 'user' | 'assistant' | 'tool'
  ordinal       INTEGER NOT NULL,          -- monotonic per session
  blocks_json   TEXT NOT NULL,             -- typed sub-blocks (text, products, comparison, outfit...)
  tool_name     TEXT,                      -- only set when role='tool'
  tool_call_id  TEXT,                      -- correlation with the assistant tool_calls entry
  created_at    TEXT NOT NULL
);

CREATE INDEX idx_messages_session_ordinal ON messages(session_id, ordinal);

CREATE TABLE shortlists (
  session_id    TEXT NOT NULL REFERENCES sessions(id) ON DELETE CASCADE,
  product_id    TEXT NOT NULL,
  lane          TEXT NOT NULL,             -- 'love' | 'maybe' | 'skip'  (UX move #1)
  product_snapshot_json TEXT NOT NULL,     -- so the share page still works after MCP delists
  added_at      TEXT NOT NULL,
  PRIMARY KEY (session_id, product_id)
);

CREATE TABLE saved_outfits (
  id            TEXT PRIMARY KEY,
  session_id    TEXT NOT NULL REFERENCES sessions(id) ON DELETE CASCADE,
  anchor_product_id TEXT NOT NULL,
  items_json    TEXT NOT NULL,             -- array of normalized products  (UX move #4)
  saved_at      TEXT NOT NULL
);
```

UX-move provenance for these columns:

- `sessions.view_mode` exists because of move #1 (collage toggle persisted per session).
- `preferences.source` exists because of move #3 — the PreferencesCard needs to render *why* a preference is on file ("you said so" vs "we inferred from your last query").
- `shortlists.product_snapshot_json` exists because of moves #1 and #7 — share pages must still render even if a merchant delists.
- `saved_outfits.items_json` exists because of move #4.
- `sessions.summary_blob` exists because of move #7.
- `messages.blocks_json` exists because every assistant message is a list of typed sub-blocks (generative UI), not a string.

What is **not** modeled and why:

- No `users` table. Sessions are anonymous; the cookie is the identity. [DEFERRED — auth, accounts]
- No `merchants` table. Merchant trust signals are denormalized into `messages.blocks_json` and `shortlists.product_snapshot_json`. We do not own a merchant directory and don't want to pretend we do.
- No `events`/analytics table on the hot path. PostHog or Plausible covers this without us building a pipeline. [ASSUMPTION — we wire analytics in Cycle 6.]

## 5. Tool contract

Every tool exports a single object conforming to:

```ts
interface Tool<TArgs, TResult> {
  name: string;                       // snake_case, matches Groq tool name
  description: string;                // one paragraph; the LLM reads this
  parameters: JSONSchema7;            // standard JSON Schema, drafted for Groq's OpenAI-shape
  emits: ServerEventType[];           // which FE event types this tool's results map to
  execute(args: TArgs, ctx: ToolContext): Promise<TResult>;
}

interface ToolContext {
  sessionId: string;
  log: FastifyBaseLogger;
  emit: (event: ServerEvent) => void; // tools may emit progress events directly
  preferences: PreferencesSnapshot;
  cache: Cache;
  signal: AbortSignal;                // honoured if the FE disconnects
}
```

`toolRegistry.toGroqSchema()` produces the OpenAI-shape `tools[]` array. `toolRegistry.dispatch(name, args, ctx)` validates args with the JSON schema, runs `execute`, and converts the result into one or more `ServerEvent`s before returning the assistant-visible string for the next turn.

The full registry by end of build:

| Tool                        | Description                                              | Cycle | Emits                                    |
|-----------------------------|----------------------------------------------------------|-------|------------------------------------------|
| `search_catalog`            | Free-form product search; the workhorse.                 | 1     | `products`, `tool_status`                |
| `get_product_details`       | Fetch full record + merchant block for one id.           | 1     | `products` (single), `tool_status`       |
| `compare_products`          | Side-by-side compare of N product ids.                   | 1     | `comparison`, `tool_status`              |
| `save_preference`           | Persist a user preference (size, budget, palette, ...).  | 2     | `preference_update`, `tool_status`       |
| `get_preferences`           | Read current preferences.                                | 2     | (assistant-visible only, no FE event)    |
| `recommend_outfit`          | Build a coordinated bundle around an anchor product.     | 3     | `outfit`, `tool_status`                  |
| `extract_style_from_image`  | Vision: image URL → attributes + suggested query.        | 4     | `moodboard`, `tool_status`               |

Each tool may **also** be invoked outside the LLM loop for deterministic FE actions (e.g. PreferencesCard PUT goes straight to `savePreference.execute` — bypassing Groq). The contract is reused; the dispatcher just runs without the LLM.

## 6. Streaming protocol

SSE on `POST /api/chat`. Each event line set is one frame:

```
event: <type>
data: <json>

```

Frames are flushed immediately; no batching. Heartbeat `event: ping` every 15 s to keep proxies happy.

All event payloads are Zod-validated server-side before write. Frontend re-validates on receive (cheap; protects against future drift).

```ts
// stream/events.ts (TypeScript reference; full Zod schemas in code)

type ServerEvent =
  | { type: 'text_delta';        text: string }
  | { type: 'tool_status';       toolCallId: string;
                                 name: string;
                                 args?: unknown;
                                 status: 'running' | 'done' | 'error';
                                 errorMessage?: string }
  | { type: 'products';          toolCallId: string;
                                 query: string;
                                 products: NormalizedProduct[] }   // includes reasoningChips, merchantInfo
  | { type: 'comparison';        toolCallId: string;
                                 products: NormalizedProduct[];
                                 axes: string[] }                  // e.g. ['price','rating','shipping']
  | { type: 'moodboard';         toolCallId: string;
                                 imageUrl: string;
                                 description: string;
                                 attributes: string[];
                                 suggestedQuery: string }
  | { type: 'reasoning_chip';    productId: string;
                                 chip: ReasoningChip }             // rare: late-binding chips
  | { type: 'preference_update'; key: string; value: unknown; source: 'user'|'inferred'|'agent' }
  | { type: 'outfit';            toolCallId: string;
                                 anchorProductId: string;
                                 items: NormalizedProduct[];
                                 rationale: string }
  | { type: 'error';             code: 'rate_limited' | 'mcp_error' | 'tool_error'
                                       | 'invalid_request' | 'internal';
                                 message: string;
                                 retryable: boolean }
  | { type: 'done';              turnsUsed: number };
```

**`reasoning_chip` rationale.** Chips are usually attached to products inside the `products` event. The standalone `reasoning_chip` event exists for the case where chip computation is expensive (e.g. a follow-up MCP call to fetch shipping times) and we want to render the card immediately and decorate it asynchronously. [ASSUMPTION — keep this in the protocol from day one; cheaper to deprecate later than to add later.]

**Client behaviour.**
- Use `@microsoft/fetch-event-source` (POST + SSE).
- On `error: rate_limited` → retry once with exponential backoff (5s, 15s, give up).
- On network drop mid-stream → DO NOT auto-reconnect; resuming a half-finished agent turn is incorrect. Surface a "connection lost, retry?" affordance and start a fresh request. ADR-0002 explains.
- On `done` → finalize the message and PUT to `/api/session/:id/messages` to persist.

## 7. Failure modes & resilience

| Failure                                    | What we do                                                                                                                                                                   |
|--------------------------------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **Groq 429 (rate limit)**                  | `groqClient` retries once with jittered backoff. If still 429, switch model `llama-3.3-70b-versatile` → `llama-3.1-8b-instant` for *this turn only* — this rescues per-minute RPM bursts. On Groq account-level daily-quota exhaustion (RPD/TPD), the `llama-3.1-8b-instant` fallback model also 429s — daily exhaust isn't a "few-second" retry. `/api/chat` returns `error code=rate_limited retryable=true`. The user-facing copy ("Hitting traffic — retrying in a few seconds") is intentionally optimistic; daily exhaust is not. A banner SSE arm that shifts copy after N consecutive 429s in M seconds is a Stage-2 add. |
| **Groq 5xx or stream stall**               | Abort the stream, emit `error: internal`. No auto-retry inside the agent loop — the user kicks it off again. Logged with full request id.                                    |
| **Catalog MCP 429**                        | `mcpClient` already retries 429/5xx with backoff (2 retries). If still failing, the tool returns `{ products: [], error: 'catalog_unavailable' }`. Agent receives the tool result, can recover ("I couldn't reach Shopify, try in a minute?"). |
| **Catalog MCP timeout (>10 s)**            | Hard timeout via `AbortSignal` at the `mcpClient` level. Tool returns an error result. Same recovery path as 429.                                                            |
| **Tool returns malformed data**            | Each tool's result is Zod-validated before being converted to events. Validation failure emits `tool_status: error` + `error: tool_error`. The assistant turn gets a synthesized failure string in tool-message form so the loop can recover next turn. |
| **SSE disconnect mid-stream**              | Server-side: detect via `request.raw.on('close')` → abort the in-flight Groq stream and any pending MCP calls (`AbortController`). Mark the assistant message as `truncated` in `messages.blocks_json`. Client-side: no auto-reconnect (see ADR-0002). |
| **SQLite locked / write contention**       | `better-sqlite3` is synchronous and single-process — locking is rare. WAL mode handles concurrent reads. Writes are serialized through repo functions; if one throws SQLITE_BUSY we retry once after 25 ms then surface a 503. Postgres migration removes this concern. |
| **User uploads non-image / huge file**     | `@fastify/multipart` limits to 8 MB. Magic-byte sniff via `file-type` to confirm `image/jpeg|png|webp`. Reject everything else with 415. [DEFERRED — virus scanning via ClamAV; revisit if we ever store uploads beyond 24h.] |
| **Image URL hostile (SSRF via vision)**   | Vision tool only accepts URLs we minted from `/api/upload` (signed with HMAC, 24h expiry). It refuses external URLs. ADR-0003 reasoning applied here.                       |
| **Tool args from LLM are nonsense**        | JSON-schema validation in the dispatcher returns a structured error to the LLM as the tool message; the agent loop can re-issue with corrected args. Capped at 4 turns total. |
| **Infinite tool-call loop**                | Hard cap: 4 agent turns per chat request. After turn 4, force `tool_choice: 'none'` for one final turn to extract a textual answer, then `done`.                              |
| **Cost runaway / abuse**                   | Per-IP rate limit on `/api/chat` via `@fastify/rate-limit` (10 req/min, 200 req/day). Per-session message count cap (200/day). Both enforced at the route level.             |

What we explicitly **do not** do: in-memory state for active sessions on the server. Every turn's state must be reconstructible from SQLite + the request body. This lets us restart the backend without breaking active conversations once they retry.

## 8. Scaling story

The path is intentional: optimize for time-to-launch, document the upgrade triggers, do not pre-build them.

**Stage 1 — today (0 to ~1k cumulative sessions).**
- Single Fly.io machine, 1 shared vCPU, 512 MB.
- SQLite on a Fly volume (1 GB).
- Groq free tier.
- In-process LRU cache.

Triggers to move on: SQLite file > 200 MB, or daily Groq requests > 10K, or p95 SSE latency > 4 s.

**Stage 2 — ~1k MAU.**
- **DB**: SQLite → libSQL (Turso). Same `better-sqlite3`-shaped API via `@libsql/client`. Repos are the only files that change. Replicate to a read replica near the FE region.
- **Cache**: same LRU, but TTL drops to 5 min. No Redis yet.
- **Groq**: pay-as-you-go on the same models. The `llama-3.1-8b-instant` fallback path stays.
- **Compute**: scale Fly to 2 machines, sticky sessions via Fly's `fly-replay` header so an in-flight SSE keeps hitting the machine that started it. **No horizontal scaling of SQLite writes** — libSQL gives us this for free.

**Stage 3 — ~100k MAU.**
- **DB**: libSQL → Postgres (Neon or Supabase). Repos abstract this cleanly; only `db/sqlite.ts` changes module. ADR-0004 prefigures the migration.
- **Cache**: introduce Redis (Upstash) for cross-process catalog cache; LRU stays as L1.
- **Queue**: introduce a worker queue (BullMQ on Redis) **only** for slow, non-user-facing tasks: OG image pregeneration, share-page hydration, abuse classification. The agent loop itself stays in-process — it's interactive and benefits from co-location.
- **Compute**: Fly autoscaling group (~5–10 machines) + a separate worker pool. CDN in front for `/s/[id]`.

[ASSUMPTION — Vercel for the FE remains correct through all three stages. The backend is the thing that moves.]

## 9. Security posture

**Secrets.**
- `GROQ_API_KEY`, `SHOPIFY_CLIENT_SECRET`, and DB credentials live **only** in the backend env. The frontend has zero references; `next.config.mjs` only proxies `/api/*`.
- Local dev: `.env` files, gitignored. Production: Fly secrets / Vercel env.

**CORS.**
- `ALLOWED_ORIGINS` whitelist. Default to the deployed FE origin only. No `*`.
- SSE responses set `Access-Control-Allow-Origin` from this list; credentialled requests not used (sessions are cookie-based, same-origin via the Next.js rewrite proxy, so CORS preflight is not on the hot path).

**Cookies.**
- `agentic_sid`: HTTP-only, Secure, SameSite=Lax, 30-day expiry. Holds the session id only. Rotated on demand.
- No CSRF token because writes are JSON with `content-type: application/json` from same-origin via the Next.js proxy. If we ever expose a third-party origin, add CSRF.

**Rate limiting.**
- `@fastify/rate-limit` on all `/api/*` routes. `/api/chat` 10/min/IP; `/api/upload` 5/min/IP; `/api/session/*` 60/min/IP. Memory store at Stage 1, Redis-backed at Stage 2+.

**Uploads.**
- 8 MB max. Magic-byte sniff (`file-type`). MIME allowlist: `image/jpeg`, `image/png`, `image/webp`. Files stored under `UPLOAD_DIR` with random nanoid filenames. Served via the backend (not directly from disk) with a signed-URL handler so we can revoke. 24h purge cron. [DEFERRED — virus scan / NSFW classification. Revisit if we open ingestion to a real user base.]
- **Upload-signing key separation (Cycle 6).** Signed upload URLs are HMAC'd with a dedicated `UPLOAD_SIGNING_SECRET`, not the `IP_HASH_SALT` (which Cycle 4 reused). In production this env var is **required** (a Zod refine in `config/env.ts` throws on boot if it's missing). In `development`/`test`, it falls back to `IP_HASH_SALT` with a single `[env] UPLOAD_SIGNING_SECRET not set; falling back to IP_HASH_SALT for dev` warning at boot. This keeps the dev ergonomics from Cycle 4 while making the prod surface explicit — and means rotating the upload key never accidentally invalidates session IP hashes (or vice versa).

**CSP.**
- `default-src 'self'; img-src 'self' https: data:; connect-src 'self' https:; script-src 'self' 'unsafe-inline' 'unsafe-eval' (dev only — Next.js dev); style-src 'self' 'unsafe-inline'; font-src 'self' data:; frame-ancestors 'none'; base-uri 'self'`.
- In prod, drop `unsafe-eval` and replace `unsafe-inline` for scripts with a nonce via Next.js middleware. [DEFERRED to Cycle 6 hardening.]
- `Strict-Transport-Security: max-age=63072000; includeSubDomains; preload` on the backend response too.

**Abuse mitigations.**
- Per-session daily message cap (200) to bound Groq spend.
- IP hash in `sessions.ip_hash` for forensic dedup without storing raw IPs.
- Vision tool gated on backend-minted signed image URLs only (no SSRF risk).

**Session retention.**
- Anonymous sessions older than **90 days** (`sessions.updated_at < now - 90 days`) are deleted by a daily housekeeping job in `backend/src/index.ts`. Foreign-key `ON DELETE CASCADE` on `preferences`, `messages`, `shortlists`, and `saved_outfits` collapses the child rows in the same transaction. (polish-round-2 T2.5.)

**Logs.**
- Pino JSON logs. PII redaction list: image URLs (truncated), raw IPs (replaced by hash), Groq request bodies (truncated). User prompts are logged in full — they are not PII by themselves, and product debugging needs them. [ASSUMPTION — revisit if we add accounts.]

## 10. Deploy targets

**Frontend → Vercel.** Reasons in priority order:
1. Next.js 14 first-class; OG image generation via `@vercel/og` works out of the box for move #7.
2. Edge runtime for `/s/[id]` makes share-link cold starts ≈ 0.
3. Preview deployments per branch — the design lead and PO reviewers can eyeball cycles before merge.

We do **not** put the backend on Vercel. SSE on serverless functions is a 2024 footgun: long-lived streams are billed against function duration limits and most edges close them at 60–120 s. The agent loop can legitimately run 30–60 s on a complex turn.

**Backend → Fly.io** (preferred) **or Railway** (acceptable). Reasons:
1. Long-lived HTTP/SSE is a first-class citizen on a real VM, not a function.
2. Cheap persistent volumes for the SQLite file at Stage 1.
3. Fly's `fly-replay` header solves the "stick this SSE to the machine that owns the session" problem at Stage 2 without a separate session store.
4. Multi-region available when latency to Shopify Catalog MCP starts to matter.

[ASSUMPTION — we use Fly. ADR-0004 references this for the SQLite volume specifically.]
