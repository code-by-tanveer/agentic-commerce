# Architect — Ops & Scaling — Round 1

## Pre-launch blockers (HIGH)

- **No SIGTERM / graceful-shutdown handler.** `backend/src/index.ts` registers `onClose` for timers but never calls `app.close()` from a signal listener, so Fly's `kill_signal = SIGINT` / rolling-deploy `SIGTERM` causes Node to exit immediately. Every in-flight SSE is half-closed mid-stream, the agent loop never sees `AbortController.abort()`, and `db.close()` is never invoked — leaving the WAL un-checkpointed. Minimum fix: `process.on('SIGTERM', () => app.close(...))` that (a) sends `event: error code=internal retryable=true` to every open SSE, (b) waits up to `grace_period` (e.g. 25 s) for in-flight loops to drain, (c) `db.exec('PRAGMA wal_checkpoint(TRUNCATE)')` and `db.close()`, then `process.exit(0)`. Without this, every routine deploy will mid-truncate live conversations.

- **`/health` is a lie vs. docs.** `backend/src/index.ts:88` returns `{ ok: true }` unconditionally. `docs/DEPLOY.md:117` claims the endpoint also returns "a Groq-reachable boolean", and `docs/ARCHITECTURE.md:149` says the same. Fly's healthcheck therefore reports green when the backend is up but Groq creds are revoked / catalog MCP is down — exactly the failure modes a healthcheck exists to surface. Either (a) shrink the doc claim to match the code, or (b) implement a richer readiness probe (`GET /health` for Fly's liveness, `GET /ready` that hits Groq + MCP with 1 s timeouts and caches the result for 10 s). Recommend (a) for launch; defer (b) to Stage 2. Doc-vs-code drift is the more immediate blocker.

- **`UPLOAD_DIR` disk-full has no fail-safe.** `routes/upload.ts` writes to disk, then `/api/upload` returns 500 `storage_unavailable` only on `wx` write failure — but on a real ENOSPC the multipart parser may already have buffered 8 MB into memory and the volume is now wedged. There's no pre-flight check on free space, no backpressure on accept, and the 1 GB Fly volume in `docs/DEPLOY.md:62` fills at ~125 successful uploads. Combined with the 5/min/IP rate limit, three motivated IPs can fill the volume in ~10 minutes. Minimum fix: (a) on `ENOSPC`/`EDQUOT` from the write, run an emergency synchronous purge of files older than 1 hour and return 503 (not 500); (b) document a Fly volume size of ≥3 GB given DB + 24 h × peak uploads; (c) add a coarse boot-time + hourly free-space check that logs `warn` at <20% free.

- **No session/message TTL.** `sessions`, `messages`, `shortlists`, `saved_outfits` accumulate forever — Cycle 4 only purges `UPLOAD_DIR`. With `agentic_sid` cookies pinned at 30-day expiry plus PRODUCT.md's anti-account stance, every casual visitor leaves a row behind. At even a few hundred users/day this hits the Stage-2 trigger (`SQLite file > 200 MB`) inside a quarter without anyone noticing. Minimum fix: add a daily housekeeping job alongside the WAL checkpoint that deletes sessions where `updated_at < now - 90 days` (CASCADE handles messages/shortlists/outfits via the foreign keys in `0001_init.sql`). Document the 90-day window in `ARCHITECTURE.md` §9.

## Should-fix before broad rollout (MEDIUM)

- **`/api/chat` rate limit collides with CGNAT / corporate NAT.** 10 req/min/IP via `req.ip` is fine for residential broadband; a single mobile carrier CGNAT pool or office NAT shares one IP across thousands of users. The `agentic_sid` cookie is already in flight by the time the limiter runs — preferable `keyGenerator` is `cookies.agentic_sid ?? req.ip` so the limit per anonymous session, with `req.ip` only as a fallback for the very first request. This also closes a small abuse vector: an attacker can't pin everyone behind a shared NAT into the same 10/min bucket.

- **No tool-latency / Groq-duration log lines.** `Pino` setup in `index.ts` only customises `req` serialization. `ToolRegistry.dispatch` calls `tool.execute(...)` with no timing. Nothing tags `start`/`end` around `streamChatCompletion` in `agent.ts`. The Cycle 6 acceptance bar says "per-tool p95 latency dashboards (Pino logs) — every tool < 1500ms" — there's no log line to read those numbers off. Minimum fix: wrap `dispatch` with `performance.now()` deltas; emit a single `log.info({ tool, durationMs, ok }, 'tool dispatch')` per call and `log.info({ model, durationMs, promptTokens, completionTokens, turnsUsed }, 'groq chat ok')` per agent run. These two lines are the entire observability story for production triage.

- **Groq usage tagging is asymmetric.** `extractStyleFromImage.ts` passes `usageTag: 'vision'`; `agent.ts` (the hot-path text completion) passes no tag. `usage_log.jsonl` therefore differentiates only by absence-of-tag, the wrong invariant — cycle-6 explicitly called for distinct text-vs-vision tagging. One-line fix: pass `usageTag: 'text'` from `agent.ts`. The vision-usage-rate script counts on this.

- **Daily-quota fallback is not actually wired the way the doc claims.** `groqClient.ts` retries on 429/503, then attempts the fallback model — but only on a single retriable response. Groq's free-tier daily TPD/RPD exhaustion returns 429 against the *account*, not the model, so the fallback `llama-3.1-8b-instant` 429s on the very next call. The architecture doc (§7) reads as if the fallback rescues us; in practice it only rescues RPM bursts. For launch: at 14.4k req/day exhaustion every `/api/chat` returns `error code=rate_limited` until midnight UTC. Either (a) document this honestly in `ARCHITECTURE.md` §7 — "fallback handles burst 429, not daily quota; daily exhaust returns rate_limited to the user", or (b) add a circuit-breaker that, on N consecutive 429s in M seconds, surfaces a "we're seeing high traffic" banner via a new SSE event arm. Recommend (a) for launch.

- **`POST /api/session/:id/outfits` is non-idempotent.** Every retry creates a new `saved_outfits` row. If the FE retries on a transient 502 (Fly machine restart mid-deploy), the user sees their outfit duplicated. The 50-row cap papers over the worst case but doesn't fix the UX. Minimum fix: accept an optional client-generated `id` in the request body; on conflict do `INSERT OR IGNORE` and return the existing row. Same pattern as `summary` already follows.

- **`trustProxy: true` is unconditional.** `backend/src/index.ts` trusts the `X-Forwarded-For` header from any source. On Fly that's fine because Fly's proxy is the only ingress, but the moment someone exposes the backend port directly for a smoke test, an attacker can spoof `X-Forwarded-For` and circumvent every rate limit. Minimum fix: pass an explicit allowlist (`trustProxy: ['fdaa::/8']` is Fly's internal range) or, simpler, `trustProxy: 1` so only the immediate hop is trusted.

- **`IP_HASH_SALT` rotation invalidates session lookups.** `sessions.ts` and `index.ts` both feed the same salt into `sha256(ip + salt)`. Rotating the salt silently disconnects new logs from old session rows — abuse forensics breaks. Worse, `docs/DEPLOY.md` suggests `openssl rand -hex 32` at first deploy with no guidance on rotation. Minimum: add a paragraph to docs noting "IP_HASH_SALT rotation is a one-way break — forensics on pre-rotation sessions becomes impossible. Plan rotation only at user-data-purge boundaries." Optionally extend `sessions` with `salt_epoch` so two-salt windows can co-exist; defer to Stage 2.

- **Fly volume backups are not addressed in the deploy doc.** `docs/DEPLOY.md` walks volume creation but never mentions Fly's `fly volumes snapshots create` or the default 5-day automatic snapshot policy. Stage 1 has zero off-region durability — a single Fly region outage at the wrong moment loses every session row + every saved outfit + every share page since the last automatic snapshot. Minimum fix: add a "Backups" subsection documenting (a) Fly's automatic 5-day daily snapshot, (b) a manual `fly volumes snapshots create` step before any risky migration, (c) restore procedure `fly volumes create --snapshot-id <id>`. The SQLite file is small enough that a daily `litestream replicate` to S3 is overkill at Stage 1 but should be flagged as a Stage-2 trigger.

## Nice-to-have (LOW)

- **WAL checkpoint cadence is 24 h.** For a 50-concurrent-write workload that's fine; for a quiet day with one write/minute, the WAL stays small and we're effectively never checkpointing. Consider lowering to 6 h and/or triggering on `app.close()` in the SIGTERM path.
- **CORS allowlist evaluated correctly for the same-origin proxy path** — confirmed. Since FE → BE is server-side via `rewrites`, the browser never CORS-preflights `/api/*`. `ALLOWED_ORIGINS` matters only for cross-origin tooling. No fix needed; flag for the security reviewer.
- **`SseWriter` 15 s ping interval is fine for Fly's 5 min idle timeout, but no test covers the case where the FE drops without a TCP FIN** (e.g. mobile Wi-Fi → cellular handoff). Dual `close` listeners already mitigate, but worth a manual test before broad rollout.
- **No structured `usage_log.jsonl` rotation.** The file grows forever next to the DB. Add `rotate` step in the daily housekeeping job (rename → new file → keep last 7).
- **`recordUsage` in `groqClient.ts` swallows all errors.** A misconfigured `DB_PATH` or `EACCES` on the data dir means usage logging silently disappears — and the vision kill-switch is then reading from /dev/null. Log once per process when the append throws, with a circuit-breaker so we don't spam.
- **`request.id` is in the Pino serializer but not propagated to MCP / Groq call sites.** A request-id correlation across services would help triage. Defer until we have a real downstream that reads it.
- **`@fastify/rate-limit` uses in-memory store.** Single-machine Fly is correct; the moment Stage 2 lights up a second machine the rate limit is per-machine. ARCH §9 already calls out the Redis-backed store at Stage 2; ensure that switch is on the Stage-2 trigger list.

## Open ops decisions for the user

1. **Are we OK launching with no SIGTERM drain?** Every `fly deploy` rolling-restart will mid-truncate live SSEs. Low blast radius at near-zero traffic; not acceptable at launch-week scale. Recommend implementing the 25 s drain handler before rollout; if you'd rather ship and learn, accept the risk in writing.
2. **90-day session TTL — too short, too long, or do we actually want it indefinite (PRODUCT.md anti-account stance) and pay for the volume growth?** Recommend 90 days as a balance; this is a one-paragraph product call.
3. **Daily-quota narrative.** Are we comfortable with the user-visible failure mode at Groq daily exhaustion being "Hitting traffic — retrying in a few seconds" that never actually recovers until midnight UTC? If not, we need a banner SSE event arm.
4. **Volume size.** `docs/DEPLOY.md` says `--size 1` (1 GB). With the disk-full failure mode above + 90-day session retention, recommend ≥3 GB. Cheap on Fly; would rather over-provision once than fight a runtime ENOSPC.
5. **Backup posture.** Fly's 5-day automatic snapshot is the floor. Do we want a manual-snapshot-before-deploy step in the runbook, or accept "Stage 1 = best effort, no DR" formally?
